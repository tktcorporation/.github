# 観点別の問いかけリスト（PdM視点）

SKILL.md の手順で使う、各観点の具体的な問いかけ。すべてに答える必要はない。**差分と狙いを見て該当する問いを選び、答えと根拠を残す。**該当しないなら「該当なし（理由）」と書く。

すべての観点は最終的に **「ユーザーが受け取る価値がどう変わるか」「PdM として合否をどう切るか」** に接続する。技術的な記述で止めず、ユーザー体験の言葉に翻訳すること。

---

## 1. ユーザー価値の変化（➕ プラス）

「このPRでユーザーは何を得るか」を、抽象論でなく具体的な体験として書く。

- ユーザーは**何ができるようになる**か。今までできなかったことか、楽になることか
- **誰**に効くか（全ユーザー / 特定セグメント / 特定の業務をする人 / 社内オペレーター）
- 体感としてどう良くなるか（速い・分かりやすい・手間が減る・迷わない・安心できる）
- その価値は**狙ったユーザー**に届くか。届く人数 / 頻度の規模感は妥当か
- イシューが解こうとした課題が、これで実際に解消するか（部分的か、完全か）

---

## 2. ユーザー価値の変化（➖ マイナス・トレードオフ）

価値が変わる変更には、ほぼ必ず引き換えがある。「マイナスなし」と書く前に必ず疑う。

- できなくなること・制限されることはないか
- 既存ユーザーの**慣れた操作・期待**を壊さないか（ボタンの位置・文言・フローの変化）
- 手順が増える / 認知負荷が上がる / 選択肢が増えすぎて迷う、はないか
- 一部のユーザーには良いが、別の一部には悪い、という非対称はないか
- 表示・数値・状態の変化で、ユーザーが「壊れた？」「減った？」と誤解しないか
- 移行期（デプロイ直後・データ移行中）にだけ起きる不便はないか

**狙い由来のマイナスを「仕様だから」で免責しない。**「毎回表示」のように狙いそのものが生むマイナスでも、ユーザー不利益が大きければ重大度（§6）で測り直す。Major 以上なら判定に効かせ、施策の前提を疑う論点（SKILL.md Step 2.5）に昇格させる。「狙い由来だから是非を問わない」は禁句。

---

## 3. ユーザーに染み出す外部仕様

対外的な契約（API/UI/CLI/DBスキーマ/イベント/設定/エラー）のうち、**ユーザーが実際に触れて気づく部分だけ**を拾う。内部実装の都合は対象外。

- **UI / 操作**: 画面・フロー・文言・ボタン挙動の変化。ユーザーの操作習慣に影響するか
- **レスポンス / 体感速度**: 表示が速く / 遅くなるか。待ち時間や反応の変化
- **出力データ / 表示内容**: 表示される数値・件数・並び順・項目の意味が変わるか。ユーザーがそれを見て判断するなら、誤読しないか
- **エラー / 失敗時の見え方**: 失敗したときユーザーに何が見えるか。分かるか、黙って壊れるか
- **API / CLI / 連携**: それを使う外部のユーザー（開発者・連携先）にとっての契約変化。後方互換性
- **新規追加の場合**: 既存契約は壊れないが、新しい出力を取り込む下流（フロント・ローダー）との暗黙の契約が生まれる。その出力がユーザーに見えるなら正しさを確認

---

## 4. イシューの意図との整合

Step 0 で押さえた「狙い（基準線）」と、実際の変更がやっていることを突き合わせる。

- 狙い通りの価値を届けているか（一致 / 一部ズレ / 別物）
- **やりすぎ**ていないか（狙いを超えて余計な挙動変更・スコープ拡大が入っていないか）
- **やり足りない**か（狙いの一部しか満たせていない・片手落ち）
- 狙いと**違う方向**に進んでいないか（手段が目的化している、別の課題を解いている）
- イシューに書かれていない価値変化（良い / 悪い両方）が起きていないか
- 基準線の信頼度で整合判定の射程が変わる:
  - イシュー本文 / PR 説明で狙いを確認できた → 「課題との整合」まで判定してよい
  - コミットや自己申告しか無い → 「狙い通りに作れているか」までに留め、「本来の課題と合っているか」は未検証と断る
  - 記載なし → 差分から「推測した狙い」と明示した上で整合を見る

---

## 5. 想定外の変化・サイレント仕様変化

宣言された狙いに**無い**のに、ユーザーに見える形で起きる変化を能動的に探す。最も見落とされ、最も事故になりやすい。

- 狙いと無関係なのにユーザーの目に触れる挙動が変わっていないか
- デフォルト値・初期表示・並び順など、誰も宣言していないが体験に効く変化はないか
- 副作用: ある機能を直したことで、別の画面 / 別のユーザー層の体験が変わっていないか
- 削除・変更により、それに依存していたユーザーの使い方が静かに壊れないか
- 「便利になった想定外」も拾う（プラスの想定外も、意図に無いなら把握しておく価値がある）

**「想定外」に分類する前のゲート（過剰指摘を防ぐ）**:

> その変化は**ユーザーが体感できるか／不利益があるか**を必ず問う。
> - YES（ユーザーが気づく挙動が、誰も宣言せず変わった） → サイレント仕様変化として ⚠️想定外
> - NO（良性の初期化・防御的コード追加・等価な書き換えなど、体験を実質変えない／むしろ自然な変更） → ⚠️ではなく 😊良い か「該当なし」

**狙いに無い ＝ 想定外、と機械的にラベルしない。**「狙いに無い変化」のうち、ユーザー体験に意味のある差を生むものだけが想定外。良性の小変更を最重要区分に置くのは過剰指摘。

## 6. プロダクト品質リスク → ユーザー体験への影響

PdM はプロダクト品質に責任を持つ。「壊れうること」を、ユーザーがどう困るかまで翻訳して評価する。

- **どういう条件で**問題が起きるか（エッジケース・大量データ・同時操作・移行期）
- 起きたとき**誰が・どう困る**か（操作できない / 待たされる / 誤った情報を見る / データが消える）
- **検知できるか**（エラーで分かる vs サイレントに誤った結果が出る）。サイレント系は重大度を上げる
- 問題が起きたとき**切り戻せる**か、ユーザーが自衛できるか
- 影響は一時的か恒久的か、全員か一部か

**重大度の目安**（出力の判定・仕分けに使う）:

| 重大度 | 目安 |
|--------|------|
| Critical | データ破損・不可逆な損失・全ユーザー停止・誤った情報での重大な意思決定。**否** に直結 |
| Major | 多くのユーザーに見える誤動作 / 誤った数字 / 切り戻しが要る障害。マージ前に直すか合意が要る（**条件付き**） |
| Minor | 一部条件のみ・回避可能・軽微な表示崩れ。直すのが望ましいが先送り可 |

「壊れる」だけでなく「**誤った情報が静かに出る（サイレント障害）**」も含める。誤った意思決定を誘発するものは Major 以上。

**Major に上げる前の反例ガード**: それは「この種の施策が標準的に伴う既知トレードオフ」か、それとも「この実装固有の、回避可能な欠陥」か。前者（例: アカウントロックに伴うロックアウト DoS 余地、レート制限の巻き添え）は、特段に重い実害が差分から具体的に読み取れない限り**原則 Minor**。後者（例: FK 付け替え漏れ・誤統合・a11y 後退）は Major 以上。標準的トレードオフに難癖をつけて判定を重くしない。

---

## 7. 対象ユーザー・範囲（誰に・どれだけ効くか）

価値もリスクも「規模」で判断が変わる。

- この変更が効くのは全ユーザーか、特定セグメントか、ごく一部か
- 頻繁に通る導線か、めったに使われない機能か
- プラスの恩恵を受ける人数 と マイナス / リスクを被る人数 のバランス
- 規模が大きいほど、小さなマイナスでも判定に重く効く
- 事業側のプラス（転換・収益）と特定セグメントのマイナス（無料ユーザーの体験劣化等）が**別の主体**に分かれる場合、両者を同じ天秤に乗せ「誰の何を犠牲に、誰が何を得ているか」を明示する。「全無料ユーザー × 高頻度操作 × 毎回割り込み」のような最大規模 × 高頻度のマイナスは、1回あたりは軽微に見えても判定に重く効く
