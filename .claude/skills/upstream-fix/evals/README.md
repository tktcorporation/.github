# upstream-fix スキルの eval

`upstream-fix` スキルを編集したら、このディレクトリの eval を回して「狙い通り発火し・狙い通り機能する」かを確認する。スキルが特定のリポジトリに依存せず汎用に動くことを担保するのが目的。

## 検証の 2 軸

| 軸 | 何を見るか | 渡すもの |
|---|---|---|
| **Triggering（発火）** | 修正系の依頼で起動し、非修正系では起動しないか（偽陽性・偽陰性） | スキルの **description だけ** |
| **Behavior（中身）** | 起動後の診断・提案がルーブリックを満たすか | **SKILL.md 本文** + シナリオ |

description だけで発火を判定するのは、本文を見せると「使うべきか」の判断が甘くなるため。Triggering は description の品質テスト、Behavior は本文の品質テスト。

## 回し方

`scenarios.md` の各シナリオを、**新しい文脈のサブエージェント**に 1 件ずつ投げる（互いに影響させない）。

### Behavior

各シナリオで、サブエージェントに次を渡す:

1. 「`SKILL.md` を読み、このスキルに従え」
2. シナリオの【依頼】文
3. 「実ファイル編集はせず、ユーザーに返す応答だけ出せ」

返ってきた応答を、シナリオの **pass 基準** で採点する。採点は別のレビュアー（人間か別エージェント）が行う。

### Triggering

1 体のサブエージェントに description だけと候補メッセージ一覧を渡し、各メッセージを独立に `FIRE` / `NO-FIRE` 判定させる。期待値と突き合わせる。

## Pass 条件

- Triggering: 全候補で期待通り（FIRE/NO-FIRE 一致）。
- Behavior: 各シナリオの pass 基準をすべて満たす。
- いずれか落ちたら、落ちた原因を SKILL.md（description か本文）に還元し、再度回す。

## シナリオを足すとき

スキルに新しい狙い（新 Phase・新ガードなど）を入れたら、それを検証するシナリオを `scenarios.md` に追加する。シナリオは**特定のスタックに依存しない**汎用な題材で書く（特定の言語・SaaS・DB に縛らない）。
