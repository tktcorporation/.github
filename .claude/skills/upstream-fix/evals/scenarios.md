# upstream-fix eval シナリオ

回し方は [`README.md`](./README.md)。シナリオは特定のスタックに依存しない汎用な題材で書く。

## Behavior（本文の品質）

サブエージェントに `SKILL.md` を読ませ、各【依頼】への応答を生成させ、pass 基準で採点する。

### B1: 末端パッチの増殖

**依頼**: 「ある値が空だと処理が落ちる。今はその値を使う 3 箇所でそれぞれ `value || default` して凌いでる。ちゃんと直して」

**pass 基準**:
- 「同じパッチが 3 箇所」を SSOT / 正規化不足のサインと認識する
- 上流（境界で 1 度正規化、Parse, Don't Validate）での解決を提案する
- 局所修正（C）だけで済ませない。B を軸に提案する

### B2: 些末な修正（やりすぎない）

**依頼**: 「ドキュメントのタイポを 1 箇所直して。周辺に他の問題は無い」

**pass 基準**:
- 重い診断ワークフロー（A/B/C 提示）を発動しない
- 局所修正で止める。スコープを膨らませない

### B3: ゼロから再設計

**依頼**: 「ある対象の状態（複数種類）の組み合わせを if のネストで処理してて、種類が増えるたびに分岐を足してて辛い。ゼロから設計するならどうなる?」

**pass 基準**:
- Phase 3 の 4 手順（不変条件の抽出 → 本物/偽の制約の分離 → 理想形のスケッチ → 差分 = スコープ）が実際に働く
- 状態を直和型で表し、網羅的分岐（exhaustive match）で扱う具体設計に着地する
- 一気に行けない規模なら段階移行（expand-contract）に触れる

### B4: 前提が現実と食い違う

**依頼**: 「ある箇所でエラーを握りつぶしてる。直して」（ただし実コードでは、その箇所に握りつぶしは無い／指摘された仕組み自体が存在しない、という設定で検証する）

**pass 基準**:
- 症状を額面通り信じず、実コードで裏取りする
- 食い違い（報告箇所に問題が無い／前提の仕組みが無い）を、修正に入る前に伝える
- 存在しない問題を「直し」始めない

### B5: 投機的な抽象化（YAGNI）

**依頼**: 「今は実装が 1 つだけだけど、将来別のも足すかもしれないから、今のうちに抽象化レイヤーを噛ませて差し替え可能にしておきたい。やっといて」（2 つ目を足す具体的な要件は無い）

**pass 基準**:
- YAGNI の歯止めが働く
- 依頼どおりのフル抽象化（A）に飛びつかず、一段軽い方針を提案・推奨する

## Triggering（description の品質）

description だけを渡し、各メッセージを独立に `FIRE` / `NO-FIRE` 判定させる。

| ID | メッセージ | 期待 | 理由 |
|---|---|---|---|
| T1 | 「このバグ直して。ある条件で処理が落ちる」 | FIRE | 明確なバグ修正依頼 |
| T2 | 「この分岐、if のネストがひどい。リファクタリングして」 | FIRE | リファクタリング依頼 + smell |
| T3 | 「なんかバグってる気がする。原因わからんけど見て」 | FIRE | 原因不明の不具合 = 診断対象 |
| T4 | 「この設計、場当たり的だから根本から見直したい」 | FIRE | 設計の根本見直し |
| T5 | 「新しい機能を 1 から実装して。要件はこれ: ...」 | NO-FIRE | 新機能の実装（実装スキルの領域） |
| T6 | 「依存パッケージを最新に上げて。脆弱性も潰して」 | NO-FIRE | 依存更新（別スキルの領域） |
| T7 | 「この機能の仕様書を書いて」 | NO-FIRE | 文書化（別スキルの領域） |

**pass 基準**: 全 ID で期待と一致。特に T5–T7 を誤って FIRE しない（過剰発火しない）こと。
