# ドキュメントの置き場所 — コード近接を優先する

実装とドキュメントの乖離 (drift) は、**コードを見れば分かる事実を `.md` に転記したとき** に起きる。転記は必ずズレる。書く前に「この情報の SSOT はどこか」を決め、コードが SSOT なら `.md` に書かず、コードの近くに書く。

`evergreen-documentation.md` の SSOT 原則の運用版。前者が「重複させるな」、本ルールが「ではどこに置くか」を決める。

## 配置の判断表

| 情報の性質 | SSOT | 置き場所 |
|---|---|---|
| スキーマ / カラム割り当て / enum の値域 | コード | 定義の JSDoc / docstring (例: schema 定義ファイルの enum / column 定義) |
| 関数の挙動・引数・戻り値・使用上の注意 | コード | その関数の JSDoc |
| config の値・環境差分・binding | config ファイル | config / IaC / manifest ファイルのコメント |
| CLI コマンドの使い方・意図 | CLI | サブコマンドの help text |
| 設計判断の WHY (なぜこの構造か / なぜ却下したか) | ドキュメント | `docs/plans/` |
| 運用手順 (rotation / onboarding / トラブルシュート) | ドキュメント | `docs/` / `docs/runbooks/` / infra ディレクトリの README |

判断基準: **「コードや config を変えたら、この記述も直さないと嘘になるか?」** → なるなら SSOT はコード側。`.md` には書かず、コードのコメントに書いて `.md` からは 1 行で参照する。

## `.md` に書いてよいもの

- コードに表現できない **WHY** (ビジネス制約・設計判断・トレードオフ・却下した代替案の理由)
- コード外の世界の **運用手順** (外部サービス連携・secret rotation・dashboard 設定など SSOT がコードに無いもの)
- 複数コンポーネントにまたがる **アーキテクチャ図** (どれか一箇所のコードには属さない俯瞰)

## 既存ドキュメントを触るとき

- スキーマ表 / カラム表 / config 値の一覧が `.md` にあったら、コードに同じ定義があるか grep する。あれば `.md` 側を消し、コードの定義 (JSDoc 付き) を SSOT として参照する 1 行に置き換える。
- 役目を終えた設計メモ (後継ドキュメントに置換された / 不採用が確定した案) は削除する。経緯は git log が持つ。後継ドキュメント側を自己完結させてから消す (リンク切れを残さない)。
