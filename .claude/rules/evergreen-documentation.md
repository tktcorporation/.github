# Evergreen Documentation ルール

コード内コメント・JSDoc/docstring・Markdown ドキュメント（`docs/`、`.claude/rules/`、README、設計書、PR 説明、commit message を含む）すべてに適用される統一ルール。

「**今読んでも、半年後に読んでも、新しいメンバーが読んでも、誤解なく意味が通る文章**」を成立させるためのガードレール。

---

## 4 つの大原則

| 原則 | 一行サマリ |
|---|---|
| **Evergreen** | 時間が経っても腐らない。履歴ではなく現在の意図を書く |
| **Why-First** | WHAT はコードや構成図が語る。ドキュメントには WHY だけ書く |
| **SSOT** | 同じ情報を複数箇所に書かない。一箇所に書いて他は参照する |
| **Reader-Friendly** | 読者が推測しなくていい・流し読みできる・専門用語に頼らない |

---

## 原則 1: Evergreen — 時間が経っても腐らない

### 書く: 現在の WHY

- ビジネス上の制約（規約で X できない、料金プラン上 Y が必要など）
- 設計判断の意図（同期負荷を避けるため非同期化、コスト削減のため日次バッチ化など）
- 気づきにくい副作用・前提条件
- 削除・変更の判断材料（X が不要になれば削除可、Y が安定したら緩和可など）

### 書かない: 時間依存表現

| カテゴリ | 例 | なぜダメか |
|---|---|---|
| 過去との対比 | 「以前は X だったが Y に変えた」「従来は」「これまで」「legacy」 | 「以前」を知らない読者にとって意味不明 |
| 現状ハイライト | 「現在は」「今は」「目下」「現状」 | 「いつ時点の現在？」が分からなくなる |
| 将来予告 | 「今後 X する予定」「将来的に Y へ」「Phase 2 で対応」 | 予定は変わる。実装されたら陳腐化、されなければ嘘になる |
| 変更履歴 | 「2025 年 3 月に導入」「v2 で追加」「リリース X 対応」 | git log / changelog の仕事 |
| 個人名・担当 | 「〇〇さん作」「△△担当」「@user 確認済み」 | 担当は変わる。git blame で十分 |
| PR/SHA 参照 | 「PR #1234 で導入」「commit 9b4909a 参照」 | リポ移行・squash merge で死ぬリンク |
| 一時的事情 | 「テスト中なので」「今だけ」「とりあえず」 | 「とりあえず」は永久に残る |

### 判断基準

> **「履歴を知らない新メンバーが読んだとき意味が通るか？」**
>
> 通らなければ履歴情報。git log / PR 説明 / 設計ドキュメントに退避する。

### Future-proof チェック

書いた直後に「半年後の自分が読んで」誤解しないか確認する。`現在` `今` `これから` `将来` `予定` を使った文は **削除 or 言い換え** を検討:

- `現在は X を使っている` → `X を使う` （現在進行形を消すと事実宣言になる）
- `今後 Y する予定` → 削除（書いてあると「やったの？」と疑問が生じる。やる予定なら issue/plan に書く）
- `将来 Z へ移行する` → 削除 or `Z へ移行する場合は <条件>` （条件文に書き換えると evergreen になる）

## 原則 2: Why-First — Context-rich, Intent-revealing

### Explain Why, not What

WHAT（何をしているか）はコードが語る。ドキュメントには WHY（なぜそうなっているか）だけ書く。

```typescript
// ❌ Bad: WHAT を繰り返している
// ユーザー ID で fetch する
const user = await getUserById(id);

// ❌ Bad: 関数名と同じ
// returns the user
function getUser() { ... }

// ✅ Good: WHY が書いてある（コードからは読み取れない情報）
// guild owner は通常存在するが、Discord API では owner が削除済みアカウントの場合 null が返るため undefined を許容する。
const owner = guild.owner ?? null;
```

### Context-rich — 背景・制約・意図を含める

「なぜこの方法か」「なぜこの順序か」「なぜこの値か」を残す。読者が「？」となる箇所だけでよい。

| 該当場面 | 書く内容 |
|---|---|
| 公開関数 (`export`) | 目的、想定呼び出し元、使用上の注意 |
| 非自明なロジック | なぜこの方法か / なぜこの順序か |
| ワークアラウンド | 理由 + どうなれば不要になるか（条件文で書く、日付では書かない） |
| 型・インターフェース | 何を表現しているか / ライフサイクル |
| 定数 | マジックナンバーの由来 |
| 防御的デフォルト値 | なぜここでこの値か |

### Intent-revealing

意図が **名前** から分かるように書く。コメントで補わなければ意図が伝わらないなら、まず名前を直す。

```typescript
// ❌ コメントで意図を補っている
const x = 86400;  // 1日の秒数

// ✅ 名前で意図が明らかなのでコメント不要
const SECONDS_PER_DAY = 86400;
```

## 原則 3: SSOT — Single Source of Truth

### 同じ情報を複数箇所に書かない

複製は **必ずズレる**。一箇所に書いて、他からは参照する。

```markdown
❌ Bad: 同じ手順を README と CONTRIBUTING.md の両方に書く
✅ Good: README には「セットアップ手順は CONTRIBUTING.md を参照」と書く

❌ Bad: コメントに API の仕様を再記述する
✅ Good: コメントで型定義 / OpenAPI スキーマへリンクする
```

### 重複検知

新しくドキュメントを書く前に、既存のどこかに同じ情報がないか **必ず grep する**。あれば:

1. 既存の文書を更新する
2. 新しい文書からは参照する
3. どうしても両方に書く必要があるなら、片方を「正」と明示し、他方に「<元> から派生した説明」と書く

## 原則 4: Reader-Friendly — 読者の負荷を最小化

### Principle of Least Astonishment（驚き最小の原則）

**章タイトル・関数名・パラメータ名と中身を一致させる**。「インストール手順」セクションにアンインストールが書いてあると読者は驚く。

- セクション名 == セクション内容
- 関数名 == 関数の振る舞い
- 「Note:」「重要:」のラベルは本当に重要なときだけ使う（オオカミ少年化を避ける）

### Progressive Disclosure（段階的開示）

最初はシンプルに、必要に応じて詳細へ。

```markdown
✅ Good 構造:
  ## 概要 (3行)
  ## 使い方 (基本)
  ## 詳細
  ## エッジケース・トラブルシューティング

❌ Bad 構造:
  ## 詳細仕様
  ## エラーハンドリング
  ## エッジケース
  ...（基本的な使い方が出てこない）
```

冒頭 3 行で「これは何で・誰向けで・何ができるか」が伝わるか確認する。

### Plain Language

- 専門用語は **初出で説明する** か、リンクを張る
- 内輪用語・略語を多用しない（「OO 案件」「あの件」）
- 受動態より能動態（「処理される」より「X が処理する」）
- 二重否定を避ける（「無効化されていない」→「有効である」）

### Scannable Writing — 流し読みできる構造

長い段落は読まれない。**見出し・箇条書き・短文・表** で構造化する。

| 良い | 悪い |
|---|---|
| 短い段落 (3-5 文) | 10 文以上の段落 |
| 箇条書きで列挙 | コンマで延々と続く列挙 |
| 表で対比 | 「A は X、B は Y、C は Z」と散文で対比 |
| 1 文 1 主張 | 「〇〇で△△が□□し、また◇◇」 |

### Cognitive Load Reduction — 認知負荷を下げる

- **一文一意**: 一つの文に複数の主張を詰め込まない
- **文脈を明示**: 「これ」「それ」が何を指すか曖昧にしない
- **前提を明示**: 「Y を知っている前提」なら冒頭でリンクを張る
- **順序を整える**: 時系列・依存関係どおりに並べる

## レイヤー別の適用

### コードコメント (`.ts` / `.py` / `.go` 等)

- 公開関数は JSDoc / docstring で目的・背景
- 非自明なロジックには WHY の 1 行
- ワークアラウンドには条件付き寿命（「X が解決したら削除可」など、日付ではなく条件で）
- 自明なゲッター・型から明らかな関数は **コメント不要**

### Markdown ドキュメント (`.md` / `.mdx`)

- 冒頭 3 行で「何のドキュメントか」が分かるサマリ
- セクションは目的別に分け、Progressive Disclosure
- 手順は番号付きリスト、概念は箇条書き、対比は表
- 他ドキュメントへの参照は相対パスで（リンク切れ検知のため）

### PR description / commit message

- WHY を中心に書く（WHAT は diff が語る）
- 関連 issue / discussion へのリンクは OK（ここは履歴情報の正当な置き場所）

## 参考文献

- Martin Fowler: "Comments express intent at the level you cannot express in code"

- Microsoft Writing Style Guide: Plain language, scannable, future-proof
